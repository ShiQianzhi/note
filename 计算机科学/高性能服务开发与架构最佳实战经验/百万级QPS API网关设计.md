## 一、引言：百万级 QPS 意味着什么

### 1.1 数字背后的挑战

百万级 QPS（Queries Per Second，每秒查询数）不是一个简单的数字，它意味着：

- **每秒处理 1,000,000 个请求**
- **每个请求的处理时间预算约为 1 微秒**（1/1,000,000 秒）
- **每天处理约 864 亿次请求**
- **峰值流量可能是平均值的 3~10 倍**

> **直观理解**：如果一个请求是一个人，那么百万 QPS 就相当于**整个哈尔滨市的人口**在一秒钟内同时通过一个收费站。

### 1.2 为什么网关是瓶颈

API 网关作为所有流量的统一入口，天然处于系统的"咽喉"位置：

```
┌─────────┐     ┌──────────┐     ┌──────────────┐
│  客户端  │────▶│ API 网关  │────▶│  后端服务集群  │
└─────────┘     └──────────┘     └──────────────┘
                    │
                    ▼
              ┌───────────┐
              │  认证/限流  │
              │  日志/监控  │
              └───────────┘
```

所有请求都必须经过网关，因此：
- 网关挂了 = 全站挂了
- 网关慢 = 全站慢
- 网关的容量上限 = 全站的容量上限

### 1.3 设计目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| QPS | ≥ 1,000,000 | 单集群峰值吞吐 |
| P99 延迟 | ≤ 1ms | 网关自身处理延迟（不含后端） |
| 可用性 | ≥ 99.99% | 年停机时间 < 52 分钟 |
| 故障恢复 | ≤ 30s | 从故障检测到自动恢复 |

## 二、整体架构设计

### 2.1 分层架构：从 L4 到 L7

百万级 QPS 不能只靠一层网关扛，必须采用**多层级联**的架构：

```
                    ┌─────────────────────┐
                    │   DNS / GSLB 层     │  ← 全局负载均衡，机房级调度
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   L4 负载均衡层      │  ← LVS / F5，四层转发
                    │   (1000万+ QPS)     │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   L7 API 网关层      │  ← Nginx / 自研网关，七层处理
                    │   (10万/节点)       │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   后端微服务集群     │  ← 业务服务
                    └─────────────────────┘
```

**各层职责**：

| 层级 | 设备/技术 | 吞吐能力 | 主要职责 |
|------|-----------|----------|----------|
| GSLB 层 | DNS、智能DNS | 无限（理论） | 跨机房调度、就近接入、故障切流 |
| L4 层 | LVS、F5、DPDK | 1000万+/秒 | 四层转发、连接调度、无状态 |
| L7 层 | Nginx、Envoy、自研 | 10万+/节点 | 路由、认证、限流、日志、协议转换 |

> **关键原则**：能在下层解决的问题，绝不上推到上层。下层设备更简单、更快、更便宜。

### 2.2 水平扩展：Scale Out 而非 Scale Up

百万 QPS 不可能靠一台机器扛住，必须依靠**集群水平扩展**。

```
             ┌──────────┐
             │   LVS    │  ← 入口（单架 1000万 QPS）
             └────┬─────┘
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
    ┌──────┐  ┌──────┐  ┌──────┐
    │ 网关1 │  │ 网关2 │  │ ... │  ← 10个节点 × 10万 = 100万 QPS
    └──┬───┘  └──┬───┘  └──┬───┘
       │          │          │
       ▼          ▼          ▼
    ┌───────────────────────────┐
    │       后端服务集群          │
    └───────────────────────────┘
```

**扩展公式**：

```
集群总 QPS = 单节点 QPS × 节点数量 × (1 - 冗余系数)

例：100,000 × 10 × (1 - 10%) = 900,000 ≈ 100万 QPS
```

> **防御性编程提示**：永远不要让集群跑满，预留至少 30% 的冗余容量应对突发流量和节点故障。

### 2.3 无状态设计

网关节点必须是**完全无状态**的：

- ✅ 不存储会话数据
- ✅ 不依赖本地缓存（可用，但不影响正确性）
- ✅ 配置从配置中心拉取
- ✅ 任意节点可以随时下线、扩容

**为什么重要**：
- 故障时流量可以无缝切到其他节点
- 扩容时直接加机器，不用数据同步
- 发布更新不会丢请求

## 三、核心模块设计

### 3.1 请求处理流水线

一个请求在网关内部的处理流程是一个**流水线（Pipeline）**：

```
请求进入
    │
    ▼
┌──────────┐  1. 连接管理：TCP 连接复用、HTTP/2 多路复用
│ 连接层    │
└────┬─────┘
     │
     ▼
┌──────────┐  2. 协议解析：HTTP/1.1、HTTP/2、gRPC、WebSocket
│ 协议层    │
└────┬─────┘
     │
     ▼
┌──────────┐  3. 路由匹配：根据 URL、Header、Host 匹配目标服务
│ 路由层    │
└────┬─────┘
     │
     ▼
┌──────────┐  4. 前置处理：认证、鉴权、限流、参数校验
│ 前置处理  │
└────┬─────┘
     │
     ▼
┌──────────┐  5. 负载均衡：选择后端实例，转发请求
│ 转发层    │
└────┬─────┘
     │
     ▼
┌──────────┐  6. 后置处理：响应改写、日志记录、指标统计
│ 后置处理  │
└────┬─────┘
     │
     ▼
响应返回
```

> **性能关键点**：流水线中每一步都要尽可能快，尤其是前置处理中的认证和限流，必须用本地缓存 + 批量操作。

### 3.2 路由模块：百万级匹配的艺术

#### 3.2.1 路由匹配算法选型

| 算法 | 时间复杂度 | 适用场景 | 缺点 |
|------|-----------|----------|------|
| 前缀树 (Trie) | O(k)，k 为路径长度 | RESTful 路径、路径参数 | 内存占用较大 |
| 基数树 (Radix Tree) | O(k) | 路径匹配优化 | 实现复杂 |
| 哈希表 (Hash Map) | O(1) | 精确路径匹配 | 不支持路径参数 |
| 正则表达式 | O(n) | 复杂规则匹配 | 性能差，不可控 |

**推荐方案**：**精确路径用哈希，路径参数用基数树，兜底用正则**。

#### 3.2.2 路由配置优化

```
路由配置加载策略：
1. 启动时全量加载到内存
2. 配置变更时增量更新（或版本切换）
3. 读多写少，用读写锁或 Copy-On-Write
```

> **DDD 思想**：路由规则属于网关的"核心领域模型"，应该有清晰的聚合根和实体定义，避免散落在各处的 if-else。

### 3.3 认证鉴权模块

#### 3.3.1 JWT 认证的性能优化

JWT 校验是网关最常见的性能瓶颈之一，优化手段：

| 优化手段 | 效果 | 原理 |
|----------|------|------|
| Token 缓存 | 减少 90% 校验耗时 | 相同 Token 第一次校验后缓存结果 |
| 公钥缓存 | 减少网络调用 | JWKS 公钥本地缓存，定期刷新 |
| 算法选择 | 提升 5~10 倍 | RS256 → Ed25519（如果允许） |
| 批量校验 | 提升吞吐量 | 批量验证签名（特定场景） |

#### 3.3.2 多级鉴权模型

```
第0级：IP 白名单 / 黑名单        ← 最快，直接拒绝
第1级：Token 有效性校验           ← JWT 签名验证
第2级：接口级权限校验             ← 角色/权限匹配
第3级：数据级权限校验             ← 交给后端服务（网关不做）
```

> **原则**：网关只做粗粒度鉴权，细粒度的数据权限交给业务服务。网关是交通警察，不是公司前台。

### 3.4 限流模块

#### 3.4.1 限流算法对比

| 算法 | 时间/空间 | 平滑度 | 突发支持 | 分布式支持 |
|------|-----------|--------|----------|------------|
| 计数器 | O(1) / O(1) | ❌ 差 | ✅ 好 | ✅ 简单 |
| 滑动窗口 | O(n) / O(n) | ⭕ 一般 | ⭕ 一般 | ⭕ 较复杂 |
| 漏桶 | O(1) / O(1) | ✅ 好 | ❌ 差 | ⭕ 需协调 |
| 令牌桶 | O(1) / O(1) | ✅ 好 | ✅ 好 | ⭕ 需协调 |

**网关推荐**：**令牌桶算法**（支持突发流量，且实现简单）。

#### 3.4.2 多级限流架构

```
┌───────────────────────────────────────────┐
│              限流分层策略                   │
├───────────────────────────────────────────┤
│  第1层：网关本地限流（令牌桶）              │  ← 最快，抗大部分流量
│  第2层：Redis 集群限流（滑动窗口）          │  ← 精准，全局限流
│  第3层：服务端自适应限流（过载保护）        │  ← 最后一道防线
└───────────────────────────────────────────┘
```

> **防御性编程**：限流不是为了"卡"用户，而是为了"保护"系统。宁可限流，不可雪崩。

### 3.5 负载均衡模块

#### 3.5.1 负载均衡算法

| 算法 | 特点 | 适用场景 |
|------|------|----------|
| 轮询 (Round Robin) | 简单、公平 | 后端性能一致 |
| 加权轮询 | 按权重分配 | 后端性能不均 |
| 随机 | 简单，统计学上均匀 | 大量实例场景 |
| 一致性哈希 | 相同请求到相同后端 | 有状态服务、缓存场景 |
| 最少连接 | 动态感知负载 | 长连接、请求耗时不均 |
| 最快响应 (EWMA) | 基于实时延迟 | 对延迟敏感的服务 |

#### 3.5.2 健康检查

```
健康检查机制：
1. 主动检查：定期 ping / http check 后端实例
2. 被动检查：根据请求失败率自动熔断异常实例
3. 半开状态：熔断后缓慢放行少量流量试探
4. 预热机制：新加入的实例逐步增加流量
```

## 四、性能优化：从毫秒到微秒

### 4.1 网络层面优化

#### 4.1.1 内核参数调优

```
# TCP 连接相关
net.core.somaxconn = 65535          # 最大半连接队列
net.ipv4.tcp_max_syn_backlog = 65535
net.core.netdev_max_backlog = 65535

# 连接复用
net.ipv4.tcp_tw_reuse = 1           # TIME_WAIT 复用
net.ipv4.tcp_tw_recycle = 1         # 快速回收（NAT 环境慎用）

# 缓冲区
net.core.rmem_default = 262144
net.core.wmem_default = 262144
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216

# 连接保持
net.ipv4.tcp_keepalive_time = 600
net.ipv4.tcp_keepalive_intvl = 30
net.ipv4.tcp_keepalive_probes = 3
```

#### 4.1.2 连接复用

**客户端 → 网关**：
- HTTP/2 多路复用（一个 TCP 连接承载多个请求）
- WebSocket 长连接
- 连接池管理

**网关 → 后端**：
- 长连接池（预建立连接，避免三次握手）
- 连接复用（HTTP Keep-Alive）
- HTTP/2 或 gRPC（多路复用）

> **数据对比**：短连接的三次握手 + TLS 握手需要 3~4 RTT，而长连接复用只需要 0 RTT。对于 1ms 的延迟预算来说，这是生死之别。

### 4.2 内存层面优化

#### 4.2.1 零拷贝 (Zero Copy)

传统数据拷贝路径（4次拷贝）：
```
磁盘/网卡 → 内核缓冲区 → 用户空间缓冲区 → Socket 缓冲区 → 网卡
```

零拷贝（sendfile / splice）：
```
磁盘/网卡 → 内核缓冲区 → 网卡  （2次拷贝，跳过用户空间）
```

**应用场景**：
- 静态文件服务
- 反向代理（Nginx sendfile on）
- 大文件传输

#### 4.2.2 内存池与对象池

高频分配释放内存会导致：
- 内存碎片
- GC 压力（Go/Java）
- 分配器锁竞争

**解决方案**：
- 内存池：提前分配一大块内存，自己管理分配
- 对象池：sync.Pool（Go）、Apache Commons Pool（Java）
- Buffer 复用：请求/响应缓冲区循环使用

```
Go 语言示例：使用 sync.Pool 复用对象

var bufPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 0, 4096)
    },
}

func handleRequest() {
    buf := bufPool.Get().([]byte)
    defer bufPool.Put(buf[:0])  // 使用后归还，重置长度
    // 使用 buf...
}
```

### 4.3 计算层面优化

#### 4.3.1 减少锁竞争

百万 QPS 下，任何锁都是性能杀手。优化策略：

| 技术 | 原理 | 适用场景 |
|------|------|----------|
| 分片 (Sharding) | 把一个大锁拆成 N 个小锁 | 计数器、缓存 |
| 无锁数据结构 | CAS（Compare-And-Swap） | 简单计数器、链表 |
| 读写锁 | 多读少写场景 | 配置、路由表 |
| Copy-On-Write | 写时复制，读完全无锁 | 读多写极少的配置 |
| 线程本地存储 | 每个 goroutine/线程一份 | 统计数据、缓存 |

#### 4.3.2 异步化与批处理

能异步就不同步，能批量就不单条：

| 操作 | 同步方式 | 异步/批量方式 | 性能提升 |
|------|----------|---------------|----------|
| 写日志 | 每条写磁盘 | 批量刷盘 + 异步队列 | 10~100 倍 |
| 指标上报 | 每条发网络 | 本地聚合 + 批量上报 | 100+ 倍 |
| 限流计数 | 每次请求 Redis | 本地限流 + 定期同步 | 1000+ 倍 |
| 配置加载 | 每次读配置中心 | 本地缓存 + 监听变更 | 无限大 |

> **排队论直觉**：批量处理本质上是用延迟换吞吐量。批次越大，吞吐量越高，但延迟也越高。需要找到平衡点。

### 4.4 缓存策略

#### 4.4.1 多级缓存架构

```
┌─────────────────────────────────────┐
│  L1：CPU 寄存器 / L1 Cache          │  ← 纳秒级，最小
├─────────────────────────────────────┤
│  L2：进程内内存缓存                  │  ← 百纳秒级，最快
│  (Go map / sync.Map / LRU)          │
├─────────────────────────────────────┤
│  L3：共享内存 / 本地 Redis           │  ← 微秒级，本机
├─────────────────────────────────────┤
│  L4：远程 Redis 集群                │  ← 亚毫秒级，跨机器
├─────────────────────────────────────┤
│  L5：配置中心 / 数据库               │  ← 毫秒级，最慢
└─────────────────────────────────────┘
```

#### 4.4.2 缓存命中率优化

- 缓存预热：启动时预加载热门数据
- 冷热分离：热点数据放内存，冷数据放 Redis
- 缓存击穿保护：singleflight（并发请求合并）
- 缓存穿透防护：布隆过滤器、空值缓存

### 4.5 Go 语言专属优化

如果用 Go 自研网关，还有这些优化手段：

| 优化点 | 方法 | 效果 |
|--------|------|------|
| Goroutine 池 | 复用 goroutine，减少创建开销 | 降低 GC 压力 |
| 减少内存分配 | 预分配切片、复用 buffer | 减少 GC 次数 |
| 逃逸分析 | 尽量让变量在栈上分配 | 避免堆分配 |
| 调度器调优 | GOMAXPROCS、netpoller | 充分利用 CPU |
| cgo 避免 | 尽量纯 Go 实现 | 避免上下文切换开销 |
| 自定义 allocator | arena / 内存池 | 手动管理内存 |

## 五、高可用设计：永不停机

### 5.1 故障等级与影响面

| 故障等级 | 影响范围 | 恢复目标 |
|----------|----------|----------|
| P0 | 全站不可用 | 5 分钟内恢复 |
| P1 | 核心功能不可用 | 15 分钟内恢复 |
| P2 | 部分功能异常 | 1 小时内恢复 |
| P3 | 非核心功能问题 | 当天修复 |

### 5.2 多活架构

#### 5.2.1 同城双活

```
┌──────────────┐        ┌──────────────┐
│   机房 A      │        │   机房 B      │
│  (Active)    │  专线  │  (Active)    │
│  50% 流量    │◀──────▶│  50% 流量    │
└──────┬───────┘        └──────┬───────┘
       │                       │
       └───────────┬───────────┘
                   ▼
            ┌──────────────┐
            │   数据库集群   │  ← 主从同步 / 集群
            └──────────────┘
```

**特点**：
- 两个机房同时承载流量
- 任一机房故障，另一机房承接全部流量
- 延迟低（同城 RTT < 2ms）

#### 5.2.2 异地多活

```
   华北机房 (北京)          华东机房 (上海)          华南机房 (深圳)
    30% 流量                 40% 流量                 30% 流量
        │                        │                        │
        └────────────────────────┼────────────────────────┘
                                 ▼
                        ┌─────────────────┐
                        │  全局数据同步层   │
                        │  (Paxos / Raft) │
                        └─────────────────┘
```

**特点**：
- 地理容灾，应对整个城市级故障
- 数据一致性挑战大（CAP 定理）
- 成本高，一般大厂才做

### 5.3 熔断与降级

#### 5.3.1 熔断机制（熔断器模式）

```
                 失败率超过阈值
    ┌──────────┐ ──────────────▶ ┌──────────┐
    │  关闭状态  │                 │  打开状态  │  ← 直接拒绝请求
    │ (正常)    │ ◀────────────── │ (熔断)    │
    └──────────┘   冷却时间到期    └─────┬────┘
        ▲                               │
        │          半开状态：             │
        └──── 试探成功 → 关闭 ◀─── 试探 ──┘
                     试探失败 → 打开
```

**关键参数**：
- 错误率阈值（如 50%）
- 熔断时长（如 30 秒）
- 半开状态放行比例（如 10%）

#### 5.3.2 降级策略

| 降级类型 | 手段 | 示例 |
|----------|------|------|
| 功能降级 | 关闭非核心功能 | 推荐列表改用静态数据 |
| 体验降级 | 降低精度/丰富度 | 搜索结果从 20 条减到 10 条 |
| 读降级 | 读缓存/兜底数据 | 数据库挂了，读本地缓存 |
| 写降级 | 异步化 / 排队 | 写入暂时存消息队列，延后处理 |

### 5.4 平滑发布与回滚

#### 5.4.1 灰度发布（金丝雀发布）

```
流量比例：
  0% → 1% → 5% → 20% → 50% → 100%
       │    │     │      │       │
       ▼    ▼     ▼      ▼       ▼
     观察  观察   观察   观察    全量
```

**每个阶段观察指标**：
- 错误率
- 延迟（P50 / P99 / P999）
- CPU / 内存使用率
- 业务指标（订单量、用户数等）

#### 5.4.2 快速回滚

- 配置化发布：版本号管理，一键回滚
- 镜像版本：每个版本对应独立镜像
- 配置中心：秒级切换配置
- 数据库变更：兼容式变更（可回滚）

## 六、可观测性：洞察每一个请求

### 6.1 三大支柱

#### 6.1.1 Metrics（指标）

**核心指标（黄金信号）**：

| 指标 | 含义 | 关注值 |
|------|------|--------|
| 流量 (Traffic) | QPS / 并发数 | 总量、趋势 |
| 延迟 (Latency) | 请求耗时 | P50 / P95 / P99 / P999 |
| 错误率 (Errors) | 失败请求比例 | 5xx / 4xx / 超时 |
| 饱和度 (Saturation) | 资源使用率 | CPU / 内存 / 连接数 |

**推荐方案**：Prometheus + Grafana

#### 6.1.2 Logging（日志）

**网关日志分类**：

| 类型 | 内容 | 存储 |
|------|------|------|
| 访问日志 | 请求方法、URL、状态码、耗时、客户端 IP | ES / ClickHouse |
| 错误日志 | 异常栈、错误详情、请求上下文 | ES |
| 审计日志 | 关键操作、权限变更 | 独立存储（不可删除） |
| 调试日志 | 详细处理过程 | 采样 + 临时开启 |

> **性能提示**：百万 QPS 下，全量打日志会把磁盘打爆。建议：
> - 访问日志按比例采样（如 1%）
> - 错误日志全量记录
> - 异步刷盘，不要同步写磁盘

#### 6.1.3 Tracing（链路追踪）

**网关在链路中的角色**：
- 链路入口：生成 TraceID
- 跨服务传播：传递 Trace 上下文
-  Span 记录：网关自身处理耗时

**推荐方案**：Jaeger / SkyWalking / OpenTelemetry

### 6.2 告警体系

#### 6.2.1 告警分级

| 级别 | 触发条件 | 通知方式 | 响应时间 |
|------|----------|----------|----------|
| P0 紧急 | 全站不可用、错误率 > 5% | 电话 + 短信 + 群@ | 5 分钟内 |
| P1 严重 | 核心服务异常、延迟飙升 | 短信 + 群@ | 15 分钟内 |
| P2 警告 | 非核心问题、资源告警 | 群消息 | 1 小时内 |
| P3 提示 | 趋势异常、容量预警 | 邮件 / 群消息 | 当天处理 |

#### 6.2.2 避免告警风暴

- 告警收敛：同类告警合并
- 告警抑制：高级别告警抑制低级别
- 告警升级：未响应自动升级
- 静默期：已知维护期间静默

## 七、技术选型对比

### 7.1 主流网关对比

| 特性 | Nginx | Kong | APISIX | Envoy | 自研(Go) |
|------|-------|------|--------|-------|----------|
| 性能 | 极高 | 高 | 极高 | 高 | 中~高 |
| 易用性 | 中 | 高 | 高 | 中 | 低 |
| 扩展性 | Lua | 插件 | 插件 | 扩展点 | 完全可控 |
| 生态 | 极丰富 | 丰富 | 较丰富 | 云原生 | 自建 |
| 动态配置 | ❌ 需 reload | ✅ | ✅ | ✅ | ✅ |
| Service Mesh | ❌ | ❌ | ⭕ | ✅ | ⭕ |
| 适用规模 | 巨型 | 大中型 | 大中型 | 云原生 | 定制化 |

### 7.2 选型建议

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 纯流量转发、静态资源 | Nginx | 性能最高、最稳定 |
| 企业级 API 管理 | Kong / APISIX | 插件丰富、管理界面完善 |
| 云原生 / K8s 环境 | Envoy / Istio | 与 K8s 深度集成 |
| 业务高度定制化 | 自研（Go/Rust） | 完全可控、深度集成业务 |
| 百万 QPS 以上 | Nginx + LVS 前端 + 自研业务网关 | 分层架构，各层最优 |

### 7.3 为什么很多大厂最终选择自研

1. **性能极致**：通用网关为了通用性有很多额外开销，自研可以做到最小内核
2. **业务定制**：与内部系统（认证、限流、计费等）深度集成
3. **技术积累**：网关是核心基础设施，掌握在自己手里更放心
4. **成本控制**：不用付商业授权费，且可以针对硬件做极致优化
