# 限流方案设计

## 一、限流算法概述

### 1.1 常用限流算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **固定窗口计数器** | 固定时间窗口内计数 | 实现简单 | 临界问题 | 简单场景、非核心链路 |
| **滑动窗口计数器** | 时间窗口滑动，平滑流量 | 流量平滑 | 实现复杂 | 需要平滑流量的场景 |
| **令牌桶** | 匀速生成令牌，请求消耗令牌 | 允许突发流量 | 实现较复杂 | 大多数场景、推荐使用 |
| **漏桶** | 请求进入队列，匀速流出 | 流量整形 | 无法应对突发 | 流量整形、网关限流 |
| **分布式令牌桶** | 基于 Redis 的令牌桶 | 分布式场景 | 依赖 Redis | 多实例、分布式系统 |
| **自适应限流** | 根据系统负载动态调整 | 智能调节 | 算法复杂 | 复杂系统、自动化运维 |

### 1.2 核心概念

```
┌─────────────────────────────────────────────────────────────┐
│                      限流核心概念                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  • Rate Limit：速率限制（每秒/每分钟允许的请求数）           │
│  • Burst：突发允许量（超过速率限制时的缓冲）                │
│  • Window：时间窗口（计数或滑动的时间范围）                 │
│  • Key：限流维度（IP、用户ID、接口、业务类型等）            │
│  • Strategy：限流策略（拒绝、排队、降级等）                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、限流算法详解

### 2.1 固定窗口计数器

**原理**：在固定时间窗口内计数，超过阈值则拒绝请求。

```
时间轴：0----1s----2s----3s----4s----5s---->
窗口1：[====100====]
窗口2：            [====120====]
窗口3：                        [====80====]
```

**问题**：临界问题

```
时间轴：0--------1s--------2s-------->
窗口1：[=========99=========]
窗口2：                    [=========99=========]
       ↑                    ↑
       第1秒末99请求        第2秒初99请求
       总计198请求在1秒内通过！
```

**Go 实现**：

```go
package limiter

import (
	"sync"
	"time"
)

type FixedWindowLimiter struct {
	mu         sync.Mutex
	counters   map[string]int
	limit      int
	window     time.Duration
	lastReset  map[string]time.Time
}

func NewFixedWindowLimiter(limit int, window time.Duration) *FixedWindowLimiter {
	return &FixedWindowLimiter{
		counters:  make(map[string]int),
		limit:     limit,
		window:    window,
		lastReset: make(map[string]time.Time),
	}
}

func (l *FixedWindowLimiter) Allow(key string) bool {
	l.mu.Lock()
	defer l.mu.Unlock()

	now := time.Now()
	
	if last, ok := l.lastReset[key]; !ok || now.Sub(last) >= l.window {
		l.counters[key] = 0
		l.lastReset[key] = now
	}
	
	if l.counters[key] >= l.limit {
		return false
	}
	
	l.counters[key]++
	return true
}
```

### 2.2 滑动窗口计数器

**原理**：将时间窗口划分为多个小格子，滑动时只计算当前窗口内的请求数。

```
时间轴：0----0.2s----0.4s----0.6s----0.8s----1s---->
窗口大小：1秒，分为5个格子，每个格子0.2秒

t=0.9s时的窗口：
[格1] [格2] [格3] [格4] [格5]
      ←←←←←←←←←←←←←←←←←←
      当前窗口：格2+格3+格4+格5+新格

优势：避免了固定窗口的临界问题
```

**Go 实现**：

```go
package limiter

import (
	"sync"
	"time"
)

type SlidingWindowLimiter struct {
	mu           sync.Mutex
	windows      map[string][]int
	limit        int
	windowSize   time.Duration
	segmentCount int
}

func NewSlidingWindowLimiter(limit int, windowSize time.Duration, segmentCount int) *SlidingWindowLimiter {
	return &SlidingWindowLimiter{
		windows:      make(map[string][]int),
		limit:        limit,
		windowSize:   windowSize,
		segmentCount: segmentCount,
	}
}

func (l *SlidingWindowLimiter) Allow(key string) bool {
	l.mu.Lock()
	defer l.mu.Unlock()

	now := time.Now()
	segmentDuration := l.windowSize / time.Duration(l.segmentCount)
	
	if _, ok := l.windows[key]; !ok {
		l.windows[key] = make([]int, l.segmentCount)
	}
	
	segmentIndex := int(now.UnixNano() / int64(segmentDuration)) % l.segmentCount
	
	total := 0
	for i := 0; i < l.segmentCount; i++ {
		if i != segmentIndex {
			total += l.windows[key][i]
		}
	}
	
	if total >= l.limit {
		return false
	}
	
	l.windows[key][segmentIndex]++
	return true
}
```

### 2.3 令牌桶算法

**原理**：令牌桶匀速生成令牌，请求需要获取令牌才能通过，允许一定的突发流量。

```
┌─────────────────────────────────────────────────────────────┐
│                      令牌桶模型                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────┐           │
│   │              令牌桶 (容量=burst)              │           │
│   │  ┌─────┬─────┬─────┬─────┬─────┬─────┐    │           │
│   │  │  ●  │  ●  │  ●  │  ●  │  ●  │  ●  │    │           │
│   │  └─────┴─────┴─────┴─────┴─────┴─────┘    │           │
│   │       当前令牌数 = 6                        │           │
│   └─────────────────┬─────────────────────────┘           │
│                     │                                       │
│                     ▼                                       │
│   ┌─────────────────────────────────────────────┐           │
│   │   令牌生成器：每秒生成 rate 个令牌           │           │
│   │              ↑                              │           │
│   │              │ 定时任务                      │           │
│   └──────────────┴──────────────────────────────┘           │
│                                                             │
│   请求 ──▶ 获取令牌 ──▶ 通过 / 拒绝                          │
│           │                                                 │
│           └── 无令牌时等待或拒绝                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Go 实现（基于 gorilla/websocket 风格）**：

```go
package limiter

import (
	"sync"
	"time"
)

type TokenBucket struct {
	mu           sync.Mutex
	tokens       float64
	capacity     float64
	rate         float64
	lastRefill   time.Time
}

func NewTokenBucket(rate, burst float64) *TokenBucket {
	return &TokenBucket{
		tokens:     burst,
		capacity:   burst,
		rate:       rate,
		lastRefill: time.Now(),
	}
}

func (tb *TokenBucket) Allow() bool {
	tb.mu.Lock()
	defer tb.mu.Unlock()

	tb.refill()
	
	if tb.tokens >= 1 {
		tb.tokens--
		return true
	}
	return false
}

func (tb *TokenBucket) AllowN(n int) bool {
	tb.mu.Lock()
	defer tb.mu.Unlock()

	tb.refill()
	
	if tb.tokens >= float64(n) {
		tb.tokens -= float64(n)
		return true
	}
	return false
}

func (tb *TokenBucket) refill() {
	now := time.Now()
	elapsed := now.Sub(tb.lastRefill).Seconds()
	tokensToAdd := elapsed * tb.rate
	tb.tokens = min(tb.tokens+tokensToAdd, tb.capacity)
	tb.lastRefill = now
}

func min(a, b float64) float64 {
	if a < b {
		return a
	}
	return b
}
```

### 2.4 漏桶算法

**原理**：请求进入队列，按固定速率流出，起到流量整形作用。

```
┌─────────────────────────────────────────────────────────────┐
│                      漏桶模型                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   请求 ──▶ ┌───────────────────────┐ ──固定速率──▶ 处理     │
│            │       漏桶队列         │                       │
│            │   [●][●][●][ ][ ]    │                       │
│            │       容量=5          │                       │
│            └──────────┬────────────┘                       │
│                       │                                    │
│                       ▼                                    │
│                  溢出则丢弃                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Go 实现**：

```go
package limiter

import (
	"sync"
	"time"
)

type LeakyBucket struct {
	mu            sync.Mutex
	queue         chan struct{}
	rate          time.Duration
	stopChan      chan struct{}
}

func NewLeakyBucket(rate time.Duration, capacity int) *LeakyBucket {
	lb := &LeakyBucket{
		queue:    make(chan struct{}, capacity),
		rate:     rate,
		stopChan: make(chan struct{}),
	}
	
	go lb.leak()
	
	return lb
}

func (lb *LeakyBucket) Allow() bool {
	select {
	case lb.queue <- struct{}{}:
		return true
	default:
		return false
	}
}

func (lb *LeakyBucket) leak() {
	ticker := time.NewTicker(lb.rate)
	defer ticker.Stop()
	
	for {
		select {
		case <-ticker.C:
			select {
			case <-lb.queue:
			default:
			}
		case <-lb.stopChan:
			return
		}
	}
}

func (lb *LeakyBucket) Stop() {
	close(lb.stopChan)
}
```

### 2.5 分布式令牌桶（Redis + Lua）

**原理**：基于 Redis 实现分布式令牌桶，支持多实例共享限流状态。

**Lua 脚本**：

```lua
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local burst = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local window = 1000 -- 1秒窗口

local current = tonumber(redis.call("get", key) or "0")
if current < now - window then
    current = now
end

local tokens = tonumber(redis.call("hget", key .. ":tokens") or tostring(burst))
tokens = math.min(tokens + (now - current) * rate / window, burst)

if tokens >= 1 then
    tokens = tokens - 1
    redis.call("set", key, now)
    redis.call("hset", key .. ":tokens", tokens)
    redis.call("expire", key, window / 1000)
    redis.call("expire", key .. ":tokens", window / 1000)
    return 1
end

return 0
```

**Go 实现**：

```go
package limiter

import (
	"context"
	"time"

	"github.com/go-redis/redis/v8"
)

type RedisTokenBucket struct {
	client *redis.Client
	rate   int
	burst  int
}

func NewRedisTokenBucket(client *redis.Client, rate, burst int) *RedisTokenBucket {
	return &RedisTokenBucket{
		client: client,
		rate:   rate,
		burst:  burst,
	}
}

var luaScript = `
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local burst = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local window = 1000

local current = tonumber(redis.call("get", key) or "0")
if current < now - window then
    current = now
end

local tokens = tonumber(redis.call("hget", key .. ":tokens") or tostring(burst))
tokens = math.min(tokens + (now - current) * rate / window, burst)

if tokens >= 1 then
    tokens = tokens - 1
    redis.call("set", key, now)
    redis.call("hset", key .. ":tokens", tokens)
    redis.call("expire", key, window / 1000)
    redis.call("expire", key .. ":tokens", window / 1000)
    return 1
end

return 0
`

func (r *RedisTokenBucket) Allow(ctx context.Context, key string) (bool, error) {
	now := time.Now().UnixNano() / 1e6
	
	result, err := r.client.Eval(ctx, luaScript, []string{key}, r.rate, r.burst, now).Result()
	if err != nil {
		return false, err
	}
	
	return result.(int64) == 1, nil
}
```

## 三、限流维度设计

### 3.1 常见限流维度

| 维度 | 说明 | 示例 |
|------|------|------|
| **IP 限流** | 限制单个 IP 的请求频率 | 防止单个 IP 恶意攻击 |
| **用户限流** | 限制单个用户的请求频率 | 防止用户滥用 |
| **接口限流** | 限制单个接口的请求频率 | 保护核心接口 |
| **业务限流** | 按业务类型限流 | 订单创建、支付等 |
| **全局限流** | 限制整个系统的请求总量 | 保护系统整体容量 |
| **组合限流** | 多个维度组合限流 | IP + 接口、用户 + 业务 |

### 3.2 限流维度实现

```go
package limiter

import (
	"context"
	"sync"
)

type MultiDimensionLimiter struct {
	mu        sync.Mutex
	limiters  map[string]Limiter
	redisPool *redis.Client
}

type Limiter interface {
	Allow(ctx context.Context, key string) (bool, error)
}

func NewMultiDimensionLimiter(redisPool *redis.Client) *MultiDimensionLimiter {
	return &MultiDimensionLimiter{
		limiters:  make(map[string]Limiter),
		redisPool: redisPool,
	}
}

func (m *MultiDimensionLimiter) Allow(ctx context.Context, dimension, key string, rate, burst int) (bool, error) {
	m.mu.Lock()
	limiterKey := dimension + ":" + string(rune(rate)) + ":" + string(rune(burst))
	
	if _, ok := m.limiters[limiterKey]; !ok {
		m.limiters[limiterKey] = NewRedisTokenBucket(m.redisPool, rate, burst)
	}
	m.mu.Unlock()
	
	return m.limiters[limiterKey].Allow(ctx, key)
}

func (m *MultiDimensionLimiter) AllowByIP(ctx context.Context, ip string, rate, burst int) (bool, error) {
	return m.Allow(ctx, "ip", ip, rate, burst)
}

func (m *MultiDimensionLimiter) AllowByUser(ctx context.Context, userID string, rate, burst int) (bool, error) {
	return m.Allow(ctx, "user", userID, rate, burst)
}

func (m *MultiDimensionLimiter) AllowByAPI(ctx context.Context, apiPath string, rate, burst int) (bool, error) {
	return m.Allow(ctx, "api", apiPath, rate, burst)
}
```

## 四、限流策略

### 4.1 拒绝策略

```go
func rejectHandler(c *gin.Context) {
	c.JSON(http.StatusTooManyRequests, gin.H{
		"code":    429,
		"message": "Too many requests",
		"retryAfter": "10s",
	})
	c.Abort()

```

### 4.2 排队策略

```go
func queueHandler(c *gin.Context) {
	ticker := time.NewTicker(time.Millisecond * 100)
	defer ticker.Stop()
	
	for i := 0; i < 10; i++ {
		if limiter.Allow() {
			c.Next()
			return
		}
		<-ticker.C
	}
	
	rejectHandler(c)
}
```

### 4.3 降级策略

```go
func degradeHandler(c *gin.Context) {
	if limiter.Allow() {
		c.Next()
		return
	}
	
	c.JSON(http.StatusOK, gin.H{

		"message": "Service degraded",

	})

}
```

### 4.4 黑白名单策略

```go
package limiter

import (
	"sync"
)

type WhitelistLimiter struct {
	mu        sync.RWMutex
	whitelist map[string]bool
	blacklist map[string]bool
	limiter   Limiter
}

func NewWhitelistLimiter(limiter Limiter) *WhitelistLimiter {
	return &WhitelistLimiter{
		whitelist: make(map[string]bool),
		blacklist: make(map[string]bool),
		limiter:   limiter,
	}
}

func (w *WhitelistLimiter) AddWhitelist(key string) {
	w.mu.Lock()
	w.whitelist[key] = true
	w.mu.Unlock()
}

func (w *WhitelistLimiter) AddBlacklist(key string) {
	w.mu.Lock()
	w.blacklist[key] = true
	w.mu.Unlock()
}

func (w *WhitelistLimiter) Allow(ctx context.Context, key string) (bool, error) {
	w.mu.RLock()
	defer w.mu.RUnlock()
	
	if w.blacklist[key] {
		return false, nil
	}
	
	if w.whitelist[key] {
		return true, nil
	}
	
	return w.limiter.Allow(ctx, key)
}
```

## 五、Gin 中间件集成

### 5.1 基础限流中间件

```go
package middleware

import (
	"net/http"
	"strings"

	"github.com/gin-gonic/gin"
	"github.com/go-redis/redis/v8"
)

func RateLimitMiddleware(redisClient *redis.Client, rate, burst int) gin.HandlerFunc {
	limiter := NewRedisTokenBucket(redisClient, rate, burst)
	
	return func(c *gin.Context) {
		ip := getClientIP(c)
		ctx := c.Request.Context()
		
		allowed, err := limiter.Allow(ctx, ip)
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{
				"code":    500,
				"message": "Internal server error",
			})
			c.Abort()
			return
		}
		
		if !allowed {
			c.Header("Retry-After", "10")
			c.JSON(http.StatusTooManyRequests, gin.H{
				"code":    429,
				"message": "Too many requests",
			})
			c.Abort()
			return
		}
		
		c.Next()
	}
}

func getClientIP(c *gin.Context) string {
	ip := c.ClientIP()
	
	if xForwardedFor := c.GetHeader("X-Forwarded-For"); xForwardedFor != "" {
		ips := strings.Split(xForwardedFor, ",")
		ip = strings.TrimSpace(ips[0])
	}
	
	return ip
}
```

### 5.2 多维度限流中间件

```go
package middleware

import (
	"net/http"

	"github.com/gin-gonic/gin"
)

func MultiRateLimitMiddleware(limiter *MultiDimensionLimiter) gin.HandlerFunc {
	return func(c *gin.Context) {
		ctx := c.Request.Context()
		ip := getClientIP(c)
		apiPath := c.Request.URL.Path
		
		if allowed, _ := limiter.AllowByIP(ctx, ip, 100, 200); !allowed {
			c.JSON(http.StatusTooManyRequests, gin.H{
				"code":    429,
				"message": "IP rate limit exceeded",
			})
			c.Abort()
			return
		}
		
		if allowed, _ := limiter.AllowByAPI(ctx, apiPath, 1000, 2000); !allowed {
			c.JSON(http.StatusTooManyRequests, gin.H{
				"code":    429,
				"message": "API rate limit exceeded",
			})
			c.Abort()
			return
		}
		
		c.Next()
	}
}
```

## 六、分布式限流一致性保障

### 6.1 问题分析

```
分布式限流的挑战：

┌─────────┐    ┌─────────┐    ┌─────────┐
│ 实例 A  │    │ 实例 B  │    │ 实例 C  │
│         │    │         │    │         │
│ 本地计数│    │ 本地计数│    │ 本地计数│
│         │    │         │    │         │
└────┬────┘    └────┬────┘    └────┬────┘
     │              │              │
     └──────────────┼──────────────┘
                    ▼
              ┌─────────┐
              │  Redis  │
              │  共享状态│
              └─────────┘

问题：
1. 网络延迟导致状态不一致
2. Redis 宕机导致限流失效
3. 高并发下 Lua 脚本执行延迟
```

### 6.2 解决方案

**方案一：本地 + 分布式双层限流**

```go
package limiter

import (
	"context"
	"sync"
)

type HybridLimiter struct {
	mu            sync.Mutex
	localLimiters map[string]*TokenBucket
	redisLimiter  *RedisTokenBucket
	rate          int
	burst         int
}

func NewHybridLimiter(redisClient *redis.Client, rate, burst int) *HybridLimiter {
	return &HybridLimiter{
		localLimiters: make(map[string]*TokenBucket),
		redisLimiter:  NewRedisTokenBucket(redisClient, rate, burst),
		rate:          rate,
		burst:         burst,
	}
}

func (h *HybridLimiter) Allow(ctx context.Context, key string) (bool, error) {
	h.mu.Lock()
	if _, ok := h.localLimiters[key]; !ok {
		h.localLimiters[key] = NewTokenBucket(float64(h.rate), float64(h.burst))
	}
	h.mu.Unlock()
	
	if !h.localLimiters[key].Allow() {
		return false, nil
	}
	
	return h.redisLimiter.Allow(ctx, key)
}
```

**方案二：熔断器 + 限流降级**

```go
package limiter

import (
	"context"
	"sync/atomic"
	"time"
)

type CircuitBreakerLimiter struct {
	limiter       Limiter
	failureCount  int32
	successCount  int32
	threshold     int32
	halfOpenAfter time.Duration
	lastFailure   time.Time
	state         int32 // 0: closed, 1: open, 2: half-open
}

func NewCircuitBreakerLimiter(limiter Limiter, threshold int32, halfOpenAfter time.Duration) *CircuitBreakerLimiter {
	return &CircuitBreakerLimiter{
		limiter:       limiter,
		threshold:     threshold,
		halfOpenAfter: halfOpenAfter,
	}
}

func (c *CircuitBreakerLimiter) Allow(ctx context.Context, key string) (bool, error) {
	state := atomic.LoadInt32(&c.state)
	
	if state == 1 {
		if time.Since(c.lastFailure) >= c.halfOpenAfter {
			atomic.StoreInt32(&c.state, 2)
		} else {
			return true, nil
		}
	}
	
	allowed, err := c.limiter.Allow(ctx, key)
	if err != nil {
		atomic.AddInt32(&c.failureCount, 1)
		c.lastFailure = time.Now()
		
		if atomic.LoadInt32(&c.failureCount) >= c.threshold {
			atomic.StoreInt32(&c.state, 1)
			atomic.StoreInt32(&c.failureCount, 0)
		}
		
		return true, nil
	}
	
	if state == 2 {
		atomic.AddInt32(&c.successCount, 1)
		if atomic.LoadInt32(&c.successCount) >= c.threshold {
			atomic.StoreInt32(&c.state, 0)
			atomic.StoreInt32(&c.successCount, 0)
		}
	}
	
	return allowed, nil
}
```

## 十、总结

### 10.1 方案选型建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **简单场景** | 固定窗口计数器 | 实现简单，资源占用低 |
| **需要平滑流量** | 滑动窗口计数器 | 避免临界问题 |
| **大多数生产场景** | 令牌桶算法 | 支持突发流量，灵活可控 |
| **流量整形** | 漏桶算法 | 匀速流出，适合网关 |
| **分布式系统** | Redis 令牌桶 | 多实例共享状态 |
| **高可用要求** | 双层限流 + 熔断 | 本地保护 + 分布式协调 |

### 10.2 关键成功因素

1. **监控先行**：完善的指标监控和告警
2. **渐进式部署**：先本地限流，再分布式限流
3. **合理阈值**：根据业务场景设置合理的 rate 和 burst
4. **降级策略**：限流后的处理策略要明确
5. **测试验证**：充分的单元测试和压力测试

### 10.3 常见坑点

- **阈值设置过高**：限流失效
- **阈值设置过低**：正常请求被拦截
- **没有考虑突发流量**：系统在峰值时不稳定
- **Redis 单点故障**：限流依赖失效
- **缺少监控告警**：无法及时发现问题
