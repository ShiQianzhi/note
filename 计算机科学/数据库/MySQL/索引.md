## 一、什么是 MySQL 索引

索引是数据库为了加快查询速度而维护的一种数据结构。

可以把索引理解成一本书的目录：

- 没有目录时，要找某个知识点，只能从第一页翻到最后一页。
- 有目录时，可以先根据目录定位页码，再直接翻到目标位置。

在 MySQL 中也是一样。如果没有索引，查询通常需要扫描整张表：

```sql
SELECT * FROM users WHERE phone = '13800000000';
```

如果 `phone` 没有索引，MySQL 可能要一行一行比较，直到找到符合条件的数据。表只有几百行时问题不大，表有几百万、几千万行时，性能会明显下降。

如果给 `phone` 创建索引：

```sql
CREATE INDEX idx_phone ON users(phone);
```

MySQL 就可以先通过索引快速定位记录，再读取对应的数据行。

从高性能 MySQL 的角度看，索引不是“可选优化项”，而是表设计和 SQL 设计的核心。绝大多数慢 SQL，本质上都和索引设计不合理、SQL 无法使用索引、扫描行数过多有关。

## 二、索引能解决什么问题

索引主要用于减少 MySQL 需要扫描的数据量。

它能帮助解决这些问题：

- 快速定位满足条件的行；
- 减少全表扫描；
- 加速排序；
- 加速分组；
- 加速关联查询；
- 保证唯一性；
- 减少回表；
- 降低锁范围。

例如订单表：

```sql
CREATE TABLE orders (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT UNSIGNED NOT NULL,
  order_no VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  amount DECIMAL(10, 2) NOT NULL,
  created_at DATETIME NOT NULL
) ENGINE=InnoDB;
```

如果业务经常查询某个用户最近的订单：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20;
```

合适的索引通常是：

```sql
CREATE INDEX idx_user_created ON orders(user_id, created_at);
```

这样 MySQL 可以先按 `user_id` 定位用户订单，再按 `created_at` 的索引顺序读取，减少扫描和排序成本。

## 三、索引的代价

索引不是越多越好。它可以提升查询性能，但也会带来成本。

### 1. 占用磁盘空间

索引本身需要存储。表越大、索引字段越长、索引越多，占用的磁盘空间越大。

尤其在 InnoDB 中，二级索引的叶子节点会保存主键值。如果主键很长，每个二级索引都会变大。

### 2. 降低写入性能

每次执行：

```sql
INSERT
UPDATE
DELETE
```

MySQL 不仅要修改数据，还要维护相关索引。

例如一张表有 8 个索引，插入一行数据时，MySQL 可能要同时更新多个索引结构。索引越多，写入成本越高。

### 3. 增加优化器选择成本

索引太多时，优化器需要在更多执行路径中选择，有时还可能选错索引。

如果出现优化器选错索引，可能需要通过统计信息、重写 SQL、调整索引，必要时才考虑 `FORCE INDEX`。

### 4. 可能引入更多锁等待

索引设计不合理时，更新语句可能扫描大量记录，从而加大加锁范围，引发锁等待甚至死锁。

所以索引优化不是简单地“给字段加索引”，而是要围绕查询模式、写入模式、数据分布和业务一致性综合设计。

## 四、InnoDB 索引的底层结构

MySQL 中最常用的存储引擎是 InnoDB。理解 MySQL 索引，重点要理解 InnoDB 索引。

InnoDB 主要使用 B+Tree 索引。

### 1. 为什么使用 B+Tree

B+Tree 适合数据库索引，主要原因是它能减少磁盘 IO。

数据库中的数据通常以“页”为单位读取。InnoDB 默认页大小通常是 16KB。B+Tree 每个节点可以存储很多 key，所以树的高度通常很低。

例如几千万行数据的索引，B+Tree 高度可能只有 3 到 4 层。一次查询只需要读取少量页，就能定位到目标数据。

### 2. B+Tree 的特点

B+Tree 的核心特点：

- 所有数据都在叶子节点；
- 非叶子节点主要用于导航；
- 叶子节点之间通过链表连接；
- 适合等值查询；
- 适合范围查询；
- 适合排序和分页优化。

例如：

```sql
SELECT * FROM orders WHERE id BETWEEN 1000 AND 2000;
```

B+Tree 可以先定位到 `id = 1000` 附近的叶子节点，然后沿着叶子节点顺序扫描到 `id = 2000`，效率比全表扫描高很多。

### 3. 聚簇索引

InnoDB 的主键索引是聚簇索引。聚簇索引的叶子节点存放完整行数据。

这意味着 InnoDB 表的数据本身就是按主键组织的一棵 B+Tree。

例如：

```sql
CREATE TABLE users (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(64) NOT NULL,
  phone VARCHAR(20) NOT NULL
) ENGINE=InnoDB;
```

`id` 是主键，InnoDB 会按 `id` 组织整张表的数据。

实战建议：

- 每张 InnoDB 表都应该显式定义主键。
- 主键尽量短，因为二级索引会保存主键值。
- 主键尽量递增，减少页分裂和随机写。
- 不建议使用随机 UUID 作为主键。

### 4. 二级索引

除主键索引外，其他索引通常叫二级索引，也叫辅助索引。

二级索引的叶子节点存储的是：

```text
索引列值 + 主键值
```

不是完整行数据。

例如：

```sql
CREATE INDEX idx_phone ON users(phone);
```

执行：

```sql
SELECT name FROM users WHERE phone = '13800000000';
```

大致过程是：

1. 通过 `idx_phone` 找到对应的主键 id。
2. 再根据 id 回到聚簇索引读取完整行。

第二步叫回表。

## 五、索引的常见类型

### 1. 主键索引

主键索引用于唯一标识一行数据，不能为 NULL。

创建方式：

```sql
CREATE TABLE users (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  name VARCHAR(64) NOT NULL,
  PRIMARY KEY (id)
) ENGINE=InnoDB;
```

特点：

- 唯一；
- 非空；
- 一张表只能有一个主键；
- InnoDB 中主键索引就是聚簇索引。

### 2. 唯一索引

唯一索引用于保证字段值不能重复。

例如用户手机号不能重复：

```sql
CREATE UNIQUE INDEX uk_phone ON users(phone);
```

或者建表时指定：

```sql
CREATE TABLE users (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  phone VARCHAR(20) NOT NULL,
  UNIQUE KEY uk_phone(phone)
) ENGINE=InnoDB;
```

适用场景：

- 用户手机号；
- 用户邮箱；
- 订单号；
- 支付流水号；
- 第三方平台唯一 ID。

注意：MySQL 中唯一索引允许多个 NULL 值。如果业务要求唯一且必填，应把字段定义为 `NOT NULL`。

### 3. 普通索引

普通索引只用于加速查询，不保证唯一性。

```sql
CREATE INDEX idx_status ON orders(status);
```

普通索引适合高频查询条件，但要注意字段选择性。

如果 `status` 只有少量取值，例如：

- created
- paid
- shipped
- closed

那么单独给 `status` 建索引的价值可能不高。因为每个状态可能对应大量数据，扫描成本仍然很高。

更常见的做法是建立联合索引：

```sql
CREATE INDEX idx_status_created ON orders(status, created_at);
```

这样更适合查询某个状态下最近的数据：

```sql
SELECT *
FROM orders
WHERE status = 'paid'
ORDER BY created_at DESC
LIMIT 50;
```

### 4. 联合索引

联合索引是多个字段组成的索引。

```sql
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

联合索引是 MySQL 索引优化中最重要的内容之一。

它可以同时服务过滤、排序和覆盖查询。

例如：

```sql
SELECT id, amount
FROM orders
WHERE user_id = 1001
  AND status = 'paid'
ORDER BY created_at DESC
LIMIT 20;
```

索引：

```sql
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

这个索引可以先过滤 `user_id`，再过滤 `status`，并利用 `created_at` 的有序性减少排序成本。

### 5. 前缀索引

对很长的字符串字段，可以只索引前 N 个字符。

例如：

```sql
CREATE INDEX idx_email_prefix ON users(email(10));
```

前缀索引可以减少索引大小，但也有缺点：

- 不能完整区分所有值；
- 选择性可能下降；
- 通常不能用于覆盖索引；
- 前缀长度需要根据数据分布评估。

评估前缀选择性：

```sql
SELECT
  COUNT(DISTINCT email) / COUNT(*) AS full_selectivity,
  COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS prefix_selectivity
FROM users;
```

如果前 10 个字符的选择性已经接近完整字段，可以考虑前缀索引。

### 6. 全文索引

全文索引用于文本搜索。

```sql
CREATE FULLTEXT INDEX ft_title_content ON articles(title, content);
```

查询示例：

```sql
SELECT *
FROM articles
WHERE MATCH(title, content) AGAINST('mysql index');
```

注意：

- 全文索引适合关键词搜索，不适合精确事务查询；
- 中文分词效果依赖版本、配置和分词方案；
- 复杂搜索场景通常更适合 Elasticsearch、OpenSearch 等搜索引擎。

### 7. 空间索引

空间索引用于 GIS 类型字段，例如地理位置。

```sql
CREATE SPATIAL INDEX idx_location ON stores(location);
```

常见业务如果只是按经纬度范围查询，通常还会结合 geohash、经纬度普通索引或专门的地理位置服务来设计。

## 六、如何创建、查看和删除索引

### 1. 建表时创建索引

```sql
CREATE TABLE orders (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  order_no VARCHAR(64) NOT NULL,
  user_id BIGINT UNSIGNED NOT NULL,
  status VARCHAR(32) NOT NULL,
  created_at DATETIME NOT NULL,
  amount DECIMAL(10, 2) NOT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_order_no(order_no),
  KEY idx_user_created(user_id, created_at),
  KEY idx_status_created(status, created_at)
) ENGINE=InnoDB;
```

### 2. 给已有表添加索引

```sql
CREATE INDEX idx_user_created ON orders(user_id, created_at);
```

或者：

```sql
ALTER TABLE orders ADD INDEX idx_user_created(user_id, created_at);
```

添加唯一索引：

```sql
ALTER TABLE orders ADD UNIQUE KEY uk_order_no(order_no);
```

### 3. 查看索引

```sql
SHOW INDEX FROM orders;
```

或者：

```sql
SHOW CREATE TABLE orders\G
```

也可以查询系统表：

```sql
SELECT
  table_schema,
  table_name,
  index_name,
  seq_in_index,
  column_name,
  non_unique
FROM information_schema.statistics
WHERE table_schema = 'your_database'
  AND table_name = 'orders'
ORDER BY index_name, seq_in_index;
```

### 4. 删除索引

```sql
DROP INDEX idx_user_created ON orders;
```

或者：

```sql
ALTER TABLE orders DROP INDEX idx_user_created;
```

删除索引前要确认：

- 是否有 SQL 正在使用该索引；
- 是否有唯一性约束依赖该索引；
- 是否影响排序、分组、关联查询；
- 是否存在业务低峰期和回滚方案。

### 5. 修改索引

MySQL 没有直接修改索引字段的语法，一般做法是先删除旧索引，再创建新索引。

```sql
ALTER TABLE orders DROP INDEX idx_user_created;
ALTER TABLE orders ADD INDEX idx_user_status_created(user_id, status, created_at);
```

生产环境中，大表修改索引可能非常耗时，并带来锁表、IO 抖动、复制延迟等风险。需要结合 MySQL 版本、DDL 算法和业务流量评估。

## 七、联合索引与最左前缀原则

联合索引遵循最左前缀原则。

例如索引：

```sql
CREATE INDEX idx_a_b_c ON t(a, b, c);
```

这个索引可以较好支持：

```sql
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3
WHERE a = 1 AND b > 2
```

但通常不能很好支持：

```sql
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3
```

因为没有从最左列 `a` 开始使用。

### 1. 等值条件优先

对联合索引来说，等值条件通常更适合放在前面。

例如查询：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
  AND status = 'paid'
  AND created_at >= '2026-01-01';
```

可以考虑：

```sql
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

`user_id` 和 `status` 是等值过滤，`created_at` 是范围过滤。

### 2. 范围条件后的列通常无法继续用于精确定位

例如索引：

```sql
CREATE INDEX idx_user_created_status ON orders(user_id, created_at, status);
```

查询：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-01-01'
  AND status = 'paid';
```

索引可以使用 `user_id` 和 `created_at` 做范围定位，但 `status` 在范围列之后，通常不能继续用于索引精确定位，只能在扫描过程中再过滤。

因此，更合理的索引往往是：

```sql
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

### 3. 排序也要符合索引顺序

索引不仅能过滤，也能帮助排序。

例如：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20;
```

索引：

```sql
CREATE INDEX idx_user_created ON orders(user_id, created_at);
```

可以利用索引顺序减少额外排序。

如果查询改成：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY amount DESC
LIMIT 20;
```

`idx_user_created` 就无法直接满足 `amount` 排序。

## 八、覆盖索引与回表优化

覆盖索引是指查询需要的字段都能从索引中直接得到，不需要回表。

例如表：

```sql
CREATE TABLE users (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(64) NOT NULL,
  phone VARCHAR(20) NOT NULL,
  age INT NOT NULL,
  city VARCHAR(64) NOT NULL,
  KEY idx_city_age_name(city, age, name)
) ENGINE=InnoDB;
```

查询：

```sql
SELECT name
FROM users
WHERE city = '北京'
  AND age = 30;
```

如果命中 `idx_city_age_name`，索引中已经包含 `city`、`age`、`name`，就不需要回表。

`EXPLAIN` 中可能看到：

```text
Using index
```

这通常表示使用了覆盖索引。

覆盖索引适合：

- 高频查询；
- 返回字段少；
- 分页列表；
- 统计查询；
- 避免大量回表的场景。

但不要为了覆盖所有查询，把大量字段都塞进索引。索引过宽会降低写入性能，并占用更多内存和磁盘。

## 九、索引失效的常见原因

### 1. 对索引列使用函数

错误示例：

```sql
SELECT *
FROM orders
WHERE DATE(created_at) = '2026-07-22';
```

对 `created_at` 使用函数后，MySQL 可能无法利用普通索引做范围定位。

推荐改写：

```sql
SELECT *
FROM orders
WHERE created_at >= '2026-07-22 00:00:00'
  AND created_at <  '2026-07-23 00:00:00';
```

索引：

```sql
CREATE INDEX idx_created_at ON orders(created_at);
```

### 2. 对索引列做计算

错误示例：

```sql
SELECT *
FROM products
WHERE price * 100 > 10000;
```

推荐改写：

```sql
SELECT *
FROM products
WHERE price > 100;
```

### 3. 隐式类型转换

如果字段是字符串：

```sql
phone VARCHAR(20)
```

错误示例：

```sql
SELECT * FROM users WHERE phone = 13800000000;
```

推荐：

```sql
SELECT * FROM users WHERE phone = '13800000000';
```

类型不一致可能导致隐式转换，使索引无法正常使用。

### 4. 模糊查询以通配符开头

可以使用索引：

```sql
SELECT * FROM users WHERE name LIKE '张%';
```

通常难以使用普通 B+Tree 索引：

```sql
SELECT * FROM users WHERE name LIKE '%张%';
```

因为 B+Tree 索引按从左到右的顺序组织，左侧不确定时无法快速定位。

如果需要包含搜索，可以考虑全文索引或搜索引擎。

### 5. OR 条件使用不当

例如：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
   OR order_no = 'A202607220001';
```

如果 `user_id` 有索引但 `order_no` 没索引，优化器可能无法高效使用索引。

可以确保两边都有合适索引：

```sql
CREATE INDEX idx_user_id ON orders(user_id);
CREATE UNIQUE INDEX uk_order_no ON orders(order_no);
```

或者根据场景改写为 `UNION ALL`：

```sql
SELECT * FROM orders WHERE user_id = 1001
UNION ALL
SELECT * FROM orders WHERE order_no = 'A202607220001';
```

是否改写要结合重复数据和执行计划判断。

### 6. 选择性太低

例如性别字段：

```sql
gender TINYINT
```

如果只有 0 和 1 两种值，单独给 `gender` 建索引通常意义不大。因为每个值都可能匹配大量数据。

但如果和其他字段组成联合索引，可能仍然有价值：

```sql
CREATE INDEX idx_gender_created ON users(gender, created_at);
```

适用于查询某个性别下最近注册用户的场景。

### 7. 数据量太小

小表即使有索引，优化器也可能选择全表扫描。因为扫描几十行、几百行数据的成本可能比走索引再回表更低。

这不是问题。优化应该关注大表、高频 SQL、慢 SQL，而不是机械要求所有查询都走索引。

## 十、使用 EXPLAIN 分析索引

优化 SQL 前，先看执行计划。

```sql
EXPLAIN
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20;
```

常见字段：

- `type`：访问类型，反映扫描方式。
- `possible_keys`：理论上可能使用的索引。
- `key`：实际使用的索引。
- `rows`：预计扫描行数。
- `filtered`：过滤后剩余比例。
- `Extra`：额外信息，例如是否使用临时表、文件排序、覆盖索引。

### 1. type 字段

常见性能从好到差大致是：

```text
system > const > eq_ref > ref > range > index > ALL
```

重点关注：

- `const`：通过主键或唯一索引定位一行，通常很好。
- `ref`：使用非唯一索引等值查询，通常不错。
- `range`：范围扫描，常见于 `BETWEEN`、`>`、`<`、`IN`。
- `index`：扫描整个索引，比全表扫描好一些，但仍可能很慢。
- `ALL`：全表扫描，大表上要重点关注。

### 2. Extra 字段

常见值：

- `Using index`：使用覆盖索引。
- `Using where`：存储引擎返回数据后还需要过滤。
- `Using filesort`：需要额外排序，不一定真的写文件，但通常要关注。
- `Using temporary`：使用临时表，分组、排序场景要关注。
- `Using index condition`：使用索引下推。

示例：

```sql
EXPLAIN
SELECT id, amount
FROM orders
WHERE user_id = 1001
  AND status = 'paid'
ORDER BY created_at DESC
LIMIT 20;
```

如果看到 `Using filesort`，可以考虑建立：

```sql
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

再观察执行计划是否改善。

### 3. EXPLAIN ANALYZE

MySQL 8.0.18 之后支持：

```sql
EXPLAIN ANALYZE
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20;
```

它会实际执行 SQL，并返回真实执行耗时和行数信息。生产环境使用时要谨慎，尤其是对大查询和写操作相关语句。

## 十一、典型实战案例

### 1. 用户手机号登录

业务场景：

```sql
SELECT id, password_hash, status
FROM users
WHERE phone = '13800000000';
```

推荐索引：

```sql
CREATE UNIQUE INDEX uk_phone ON users(phone);
```

原因：

- 手机号业务上应唯一；
- 唯一索引可以快速定位；
- 还能防止重复注册。

如果要减少回表，可以根据查询字段考虑：

```sql
CREATE UNIQUE INDEX uk_phone_login ON users(phone, id, password_hash, status);
```

但是否值得这样做，要看登录查询频率、字段长度、写入成本和安全策略。通常不建议把过长字段盲目放入索引。

### 2. 订单列表分页

业务场景：

```sql
SELECT id, order_no, amount, status, created_at
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20 OFFSET 10000;
```

推荐索引：

```sql
CREATE INDEX idx_user_created ON orders(user_id, created_at);
```

但深分页仍然有问题。`OFFSET 10000` 意味着 MySQL 需要跳过前 10000 行。

更好的方式是基于游标分页：

```sql
SELECT id, order_no, amount, status, created_at
FROM orders
WHERE user_id = 1001
  AND created_at < '2026-07-22 12:00:00'
ORDER BY created_at DESC
LIMIT 20;
```

如果 `created_at` 可能重复，可以使用组合游标：

```sql
SELECT id, order_no, amount, status, created_at
FROM orders
WHERE user_id = 1001
  AND (
    created_at < '2026-07-22 12:00:00'
    OR (created_at = '2026-07-22 12:00:00' AND id < 900000)
  )
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

对应索引：

```sql
CREATE INDEX idx_user_created_id ON orders(user_id, created_at, id);
```

### 3. 库存扣减

业务场景：

```sql
UPDATE stock
SET quantity = quantity - 1
WHERE product_id = 1001
  AND quantity > 0;
```

推荐索引：

```sql
CREATE UNIQUE INDEX uk_product_id ON stock(product_id);
```

原因：

- `product_id` 唯一定位库存记录；
- 更新时锁范围小；
- 避免扫描大量库存行。

不要先查库存再更新：

```sql
SELECT quantity FROM stock WHERE product_id = 1001;
UPDATE stock SET quantity = quantity - 1 WHERE product_id = 1001;
```

高并发下这种写法容易出现并发问题。更推荐使用带条件的原子更新。

### 4. 多租户数据查询

业务场景：

```sql
SELECT *
FROM documents
WHERE tenant_id = 10
  AND status = 'published'
ORDER BY updated_at DESC
LIMIT 50;
```

推荐索引：

```sql
CREATE INDEX idx_tenant_status_updated ON documents(tenant_id, status, updated_at);
```

多租户系统中，`tenant_id` 通常应该放在很多联合索引的前面，因为绝大多数查询都要限制租户范围。

### 5. 统计某天订单数

错误写法：

```sql
SELECT COUNT(*)
FROM orders
WHERE DATE(created_at) = '2026-07-22';
```

推荐写法：

```sql
SELECT COUNT(*)
FROM orders
WHERE created_at >= '2026-07-22 00:00:00'
  AND created_at <  '2026-07-23 00:00:00';
```

索引：

```sql
CREATE INDEX idx_created_at ON orders(created_at);
```

如果经常按状态和日期统计：

```sql
SELECT COUNT(*)
FROM orders
WHERE status = 'paid'
  AND created_at >= '2026-07-22 00:00:00'
  AND created_at <  '2026-07-23 00:00:00';
```

可以考虑：

```sql
CREATE INDEX idx_status_created ON orders(status, created_at);
```

### 6. 关联查询优化

业务场景：

```sql
SELECT o.id, o.order_no, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'paid'
  AND o.created_at >= '2026-07-01'
ORDER BY o.created_at DESC
LIMIT 100;
```

推荐索引：

```sql
CREATE INDEX idx_status_created_user ON orders(status, created_at, user_id);
```

`users.id` 通常已经是主键，无需额外创建。

关联查询的索引原则：

- 被驱动表的关联字段要有索引；
- 过滤条件和排序字段要结合访问路径设计；
- 不要让大表无索引关联大表；
- 先用 `EXPLAIN` 判断驱动顺序和扫描行数。

## 十二、索引设计原则

### 1. 根据查询设计索引

不要看到字段就加索引。应该先列出高频 SQL：

- 查询条件是什么；
- 是否有排序；
- 是否有分组；
- 返回字段有哪些；
- 每次查询大约命中多少行；
- 查询频率有多高；
- 写入频率有多高。

然后围绕这些 SQL 设计索引。

### 2. 高选择性字段更适合建索引

选择性指字段值的区分度。

可以用下面 SQL 粗略判断：

```sql
SELECT COUNT(DISTINCT user_id) / COUNT(*) AS selectivity
FROM orders;
```

结果越接近 1，选择性越高。选择性越高，索引过滤效果通常越好。

### 3. 联合索引优于多个单列索引

对于查询：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
  AND status = 'paid';
```

相比两个单列索引：

```sql
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);
```

很多时候更适合一个联合索引：

```sql
CREATE INDEX idx_user_status ON orders(user_id, status);
```

联合索引可以更精确地缩小扫描范围。

### 4. 避免重复索引

例如已经有：

```sql
CREATE INDEX idx_user_status ON orders(user_id, status);
```

再创建：

```sql
CREATE INDEX idx_user ON orders(user_id);
```

多数情况下 `idx_user` 是冗余的，因为 `idx_user_status` 的最左前缀已经包含 `user_id`。

但如果 `idx_user_status` 很宽、`idx_user` 被极高频查询使用，也可能有保留价值。是否删除要结合执行计划和实际负载判断。

### 5. 控制索引数量和宽度

索引不是免费资源。

建议：

- 单表索引数量不要无节制增长；
- 联合索引字段不要过多；
- 避免把大字段放入索引；
- 优先索引过滤、排序、关联最关键的字段；
- 定期清理长期不用的索引。

### 6. 索引字段尽量保持稳定

频繁更新的字段如果放在多个索引中，会带来额外维护成本。

例如订单状态 `status` 会变更，但通常变更次数有限，可以接受。

但如果某个字段每秒频繁更新，并且被多个索引引用，就要评估写入压力。

## 十三、索引与锁的关系

InnoDB 的行锁通常是加在索引记录上的。索引设计会直接影响锁范围。

### 1. 有索引时锁范围更小

```sql
UPDATE orders
SET status = 'closed'
WHERE order_no = 'A202607220001';
```

如果 `order_no` 有唯一索引：

```sql
CREATE UNIQUE INDEX uk_order_no ON orders(order_no);
```

这条更新通常只需要锁定对应订单。

### 2. 没有索引时可能扫描大量记录

如果 `order_no` 没有索引，MySQL 可能扫描大量记录。扫描过程中会造成更大的锁范围和更多锁等待。

### 3. 范围查询可能加范围锁

在 `REPEATABLE READ` 隔离级别下，InnoDB 为了防止幻读，某些当前读可能使用 next-key lock。

例如：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-07-01'
FOR UPDATE;
```

如果索引是：

```sql
CREATE INDEX idx_user_created ON orders(user_id, created_at);
```

锁范围会围绕该索引范围展开。范围越大，锁冲突可能越多。

实战建议：

- 更新、删除、`SELECT ... FOR UPDATE` 必须重点关注索引；
- 尽量用唯一索引定位单行；
- 范围更新要分批；
- 避免在事务中执行大范围当前读。

## 十四、索引与排序、分组、分页

### 1. 排序优化

查询：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20;
```

推荐索引：

```sql
CREATE INDEX idx_user_created ON orders(user_id, created_at);
```

如果 `EXPLAIN` 中出现 `Using filesort`，说明 MySQL 需要额外排序。额外排序不一定不可接受，但大结果集排序通常要优化。

### 2. 分组优化

查询：

```sql
SELECT status, COUNT(*)
FROM orders
WHERE user_id = 1001
GROUP BY status;
```

可以考虑：

```sql
CREATE INDEX idx_user_status ON orders(user_id, status);
```

这样可以先按用户过滤，再利用索引顺序进行分组。

### 3. 分页优化

深分页问题：

```sql
SELECT *
FROM orders
ORDER BY id
LIMIT 20 OFFSET 100000;
```

MySQL 需要跳过大量数据。

优化方式：

```sql
SELECT *
FROM orders
WHERE id > 100000
ORDER BY id
LIMIT 20;
```

这种方式要求业务能接受基于游标的分页。

## 十五、索引下推

索引下推，Index Condition Pushdown，简称 ICP，是 MySQL 的一种优化。

它可以让存储引擎在扫描索引时，先判断部分条件，减少回表次数。

例如索引：

```sql
CREATE INDEX idx_name_age ON users(name, age);
```

查询：

```sql
SELECT *
FROM users
WHERE name LIKE '张%'
  AND age = 30;
```

如果使用索引下推，InnoDB 可以在索引层先判断 `age = 30`，减少不必要的回表。

在 `EXPLAIN` 的 `Extra` 中可能看到：

```text
Using index condition
```

索引下推是优化器能力，不代表可以随意设计索引。正确的联合索引顺序仍然很重要。

## 十六、不可见索引

MySQL 8.0 支持不可见索引。不可见索引不会被优化器使用，但仍会被维护。

适合在删除索引前做灰度验证。

设置索引不可见：

```sql
ALTER TABLE orders ALTER INDEX idx_user_created INVISIBLE;
```

恢复可见：

```sql
ALTER TABLE orders ALTER INDEX idx_user_created VISIBLE;
```

查看索引是否可见：

```sql
SELECT
  index_name,
  is_visible
FROM information_schema.statistics
WHERE table_schema = 'your_database'
  AND table_name = 'orders';
```

实战用法：

1. 先把疑似无用索引改为不可见。
2. 观察慢 SQL、执行计划和业务指标。
3. 如果没有问题，再删除索引。
4. 如果出现问题，快速恢复可见。

## 十七、生产环境添加索引的注意事项

### 1. 大表加索引不是小操作

对大表执行：

```sql
ALTER TABLE orders ADD INDEX idx_user_created(user_id, created_at);
```

可能带来：

- 表结构变更耗时很长；
- 大量 IO；
- 主从复制延迟；
- 元数据锁等待；
- 临时空间不足；
- 业务写入抖动。

### 2. 关注 DDL 算法

MySQL 支持不同 DDL 算法，例如 `INPLACE`、`COPY`、`INSTANT`。不同版本、不同操作支持情况不同。

可以显式指定：

```sql
ALTER TABLE orders
ADD INDEX idx_user_created(user_id, created_at),
ALGORITHM=INPLACE,
LOCK=NONE;
```

如果当前版本或操作不支持，语句可能失败。生产执行前应在测试环境验证。

### 3. 避免业务高峰执行

加索引通常应选择业务低峰期，提前评估：

- 表大小；
- 预计耗时；
- 磁盘剩余空间；
- 主从延迟；
- 是否有长事务；
- 是否有回滚方案。

### 4. 使用在线变更工具

对于超大表，可以考虑：

- pt-online-schema-change；
- gh-ost；
- 云数据库提供的在线 DDL 能力。

这些工具也不是无成本的，仍然需要压测、限速和监控。

## 十八、如何发现需要优化的索引

### 1. 慢查询日志

开启慢查询日志后，可以分析哪些 SQL 执行慢：

```sql
SHOW VARIABLES LIKE 'slow_query_log';
SHOW VARIABLES LIKE 'long_query_time';
```

慢 SQL 出现后，不要第一反应就是加索引。正确流程是：

1. 看 SQL 业务含义。
2. 看执行计划。
3. 看扫描行数。
4. 看现有索引。
5. 看数据分布。
6. 再决定是否新增或调整索引。

### 2. performance_schema

MySQL 的 `performance_schema` 可以观察 SQL 执行统计。

示例：

```sql
SELECT
  digest_text,
  count_star,
  avg_timer_wait / 1000000000 AS avg_ms,
  rows_examined_avg,
  rows_sent_avg
FROM performance_schema.events_statements_summary_by_digest
ORDER BY avg_timer_wait DESC
LIMIT 10;
```

不同 MySQL 版本字段可能略有差异，实际使用时以当前版本为准。

### 3. sys schema

MySQL 的 `sys` 库提供了一些更易读的视图。

查看冗余索引：

```sql
SELECT *
FROM sys.schema_redundant_indexes;
```

查看未使用索引：

```sql
SELECT *
FROM sys.schema_unused_indexes;
```

注意：未使用索引统计依赖运行期间采集的数据。如果数据库刚重启，或者统计周期没有覆盖完整业务周期，结果可能不准确。

## 十九、常见反模式

### 1. 每个字段都建索引

这是典型错误。索引越多，写入越慢，维护成本越高。

### 2. 只建单列索引，不建联合索引

真实业务查询通常包含多个条件。多个单列索引不一定能替代一个合适的联合索引。

### 3. 联合索引字段顺序随便放

字段顺序会影响过滤、排序、分组和范围查询。应根据 SQL 的访问模式设计。

### 4. 在大字段上盲目建索引

例如很长的 `VARCHAR`、`TEXT` 字段。应考虑前缀索引、全文索引、搜索引擎或业务侧改造。

### 5. 用 `SELECT *` 破坏覆盖索引

本来只需要几个字段，却写成：

```sql
SELECT * FROM orders WHERE user_id = 1001;
```

这会增加网络传输、回表成本和内存消耗。

更推荐只查询需要的字段：

```sql
SELECT id, order_no, status, created_at
FROM orders
WHERE user_id = 1001;
```

### 6. 忽视数据分布

同样的索引，在不同数据分布下效果完全不同。

例如 `status = 'paid'` 如果匹配 90% 的数据，索引过滤效果就很差。如果只匹配 1% 的数据，索引价值就很高。

## 二十、索引设计实战清单

建索引前：

- 这条 SQL 是否足够高频或足够慢；
- 表数据量是否已经大到需要索引；
- 查询条件、排序、分组、关联字段是什么；
- 字段选择性如何；
- 是否可以通过改写 SQL 解决；
- 是否已有索引可以复用；
- 是否会造成重复索引。

设计索引时：

- 优先考虑联合索引；
- 等值条件通常放前面；
- 范围条件通常放在等值条件后面；
- 排序字段尽量和索引顺序一致；
- 返回字段少时考虑覆盖索引；
- 控制索引字段数量和字段长度；
- 避免低选择性字段单独建索引。

上线前：

- 使用 `EXPLAIN` 验证执行计划；
- 评估加索引对写入性能的影响；
- 大表 DDL 要评估锁、IO、主从延迟和磁盘空间；
- 尽量在低峰期执行；
- 保留回滚方案。

上线后：

- 观察慢查询是否下降；
- 观察 CPU、IO、Buffer Pool、主从延迟；
- 观察锁等待和死锁；
- 检查优化器是否选择了预期索引；
- 定期清理无用索引。

## 二十一、总结

索引是 MySQL 性能优化的核心。它的目标不是“让 SQL 看起来使用了索引”，而是减少扫描行数、减少排序、减少回表、降低锁范围，并让数据库在高并发下保持稳定。

掌握 MySQL 索引，需要抓住几条主线：

- InnoDB 使用 B+Tree 组织索引；
- 主键索引是聚簇索引，决定整张表的数据组织方式；
- 二级索引保存主键值，查询完整行时可能回表；
- 联合索引遵循最左前缀原则；
- 覆盖索引可以减少回表；
- 索引会提升查询，但会增加写入和存储成本；
- SQL 写法不当会导致索引失效；
- 优化必须结合 `EXPLAIN`、慢查询、数据分布和业务访问模式。

最实用的原则是：围绕真实 SQL 设计索引，优先优化高频慢查询，用联合索引同时服务过滤、排序和覆盖查询，并在生产环境中谨慎评估每一次索引变更。
